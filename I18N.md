# Internationalization (i18n)

## Rule: No hardcoded UI strings

**All user-facing text must use i18n.** No hardcoded Turkish or English strings in components or pages.

- Use `getTranslations(namespace)` (server) or `useTranslations(namespace)` (client) from `next-intl`.
- Add every new string to **both** `messages/en.json` and `messages/tr.json`.
- Namespaces: `nav`, `common`, `hero`, `cta`, `trust`, `services`, `projects`, `about`, `contact`, `footer`, `legal`, `localeSwitcher`, `toast`.

## Adding new text

1. Add the key to `messages/en.json` and `messages/tr.json` under the right namespace.
2. In the component: `const t = useTranslations("namespace");` then `{t("key")}`.

## Routing

- `/en/*` → English UI
- `/tr/*` → Turkish UI
- Language switcher (globe) changes the locale and re-renders the whole site.

## No fallback to other locale

- Only the current locale's messages are loaded. Missing keys must not fall back to another language.
- If a key is missing in the current locale: fix the dictionary (add the key to both en.json and tr.json). Do not rely on fallback to EN when in TR or vice versa.

## Code review

When reviewing PRs, ensure no raw Turkish/English strings appear in JSX/TSX (e.g. no `"Teklif Al"` or `"Get a Quote"` in components). All such text must come from `t(...)`.

## Dev guard

In development, when locale is TR, `LocaleGuard` checks the page for common English tokens and logs a console error if any are found. Ensures TR UI stays 100% Turkish.
